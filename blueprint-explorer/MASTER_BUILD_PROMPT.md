# Master Build Prompt — Conrad Blueprint Explorer / Blue Collar Apps OS

## Mission

Build a living, governed architecture explorer for Blue Collar Apps Co. It must show how the founder operating system, Conrad / EXPO, shared services, agents, products, interfaces, workflows, documents, and infrastructure connect. The explorer is not Conrad itself. Conrad uses the explorer as a source-of-truth map and operational control surface.

## Core distinction

- **Blue Collar Apps OS:** the parent founder operating system.
- **Conrad / EXPO:** the company brain, expediter, router, guide, and orchestration layer.
- **Blueprint Explorer:** the interactive architecture and dependency map Conrad reads and updates.
- **Control Center:** the later operational cockpit built on top of the explorer.

## Required users

1. Founder: understands the whole company and issues commands.
2. Product lead: sees product boundaries, dependencies, and status.
3. Developer: sees repositories, environments, APIs, services, and technical ownership.
4. Agent operator: sees agent roles, tools, permissions, and health.
5. Reviewer: sees decisions, citations, risks, and approval history.

## Required V1 functions

1. Hierarchical navigation with all same-level systems aligned at the same level.
2. Expand/collapse architecture branches.
3. Search by product, agent, service, capability, dependency, owner, or status.
4. Filters for live, active build, platform, and concept systems.
5. Node detail panel with purpose, owner, status, capabilities, dependencies, parent, and downstream systems.
6. Export the complete architecture as JSON.
7. Responsive desktop and mobile interface.
8. Trench Design implementation using Obsidian, layered near-black surfaces, restrained Fire Orange, Bebas Neue, DM Sans, and IBM Plex Mono.
9. No invented BCA pricing, roadmap, product promises, or architecture.
10. Mark architecture-level decisions for Conrad / EXPO review.

## Required V2 functions

1. Editable nodes and relationships with role-based approvals.
2. GitHub repository links, branch status, pull-request status, and deployment state.
3. Document links and source citations from the BCA Knowledge Vault.
4. Agent registry with tools, permissions, owner, model assignments, and health.
5. Workflow visualization with triggers, approvals, retries, and run history.
6. Environment map for local, development, staging, and production.
7. Decision log with author, date, rationale, alternatives, and approval state.
8. Architecture diff view showing what changed between versions.
9. Conrad conversational command bar.
10. Audit trail for every edit and automated update.

## Required V3 functions

1. Live Control Center with product health, agent activity, incidents, launches, and decision queues.
2. Automated ingestion from GitHub, Vercel, Render, Notion, Google Drive, and approved internal services.
3. Model Router telemetry and evaluation results.
4. Cost, token, usage, and reliability reporting across approved models.
5. Dependency risk detection and orphaned-system detection.
6. Scenario mode for proposed architecture changes before approval.
7. Founder briefing generated from verified system changes.

## Governance rules

1. Conrad / EXPO is the source-of-truth coordinator.
2. Product teams may propose edits but cannot silently redefine company architecture.
3. Pricing, fleet architecture, product boundaries, shared-token policy, and roadmap commitments are architecture-level decisions.
4. CapKids requires COPPA gates and minimized child data handling.
5. AllergenShield must never guarantee allergen safety; it requires liability language, source traceability, and human confirmation.
6. Secrets remain in environment variables and never appear in the map.
7. CORS must use explicit approved origins, never wildcard origins.
8. Every automated update requires a source, timestamp, and confidence state.
9. Unknown information is labeled unknown, not guessed.
10. The product must preserve an auditable history of changes.

## Acceptance criteria

1. A user can understand the BCA hierarchy without explanation.
2. Same-level systems appear at the same visual level.
3. Every visible claim is verified, explicitly documented, inferred, or marked unknown.
4. Search and filters work without page reloads.
5. Keyboard focus is visible.
6. The app is usable on mobile.
7. Exported JSON accurately reflects the architecture data.
8. No prohibited legacy platform references exist anywhere in the project.
9. Production build completes with no errors.
10. The interface does not resemble a generic SaaS template.
